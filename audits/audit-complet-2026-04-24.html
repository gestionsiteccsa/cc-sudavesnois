2. Non-conformités RGAA / WCAG

3.2 — Contraste des textes insuffisant pour AAA
Priorité : Majeur (WCAG 2.1 AA conforme, AAA non conforme)

Les textes utilisant text-gray-600 (#4B5563) sur fond blanc ont un ratio de contraste de 6.3:1. Cela passe le seuil AA (4.5:1) mais échoue le seuil AAA (7:1) pour les textes normaux.

Éléments concernés : descriptions dans le panneau d'accessibilité, textes secondaires du footer, métadonnées.

Correction : Remplacer text-gray-600 par text-gray-700 (#374151, ratio 10.3:1) ou utiliser text-gray-600 uniquement pour les textes ≥ 18px gras / 24px normaux.
10.7 — Focus visible non optimal sur éléments spécifiques
Priorité : Mineur (AA partiellement conforme)

Plusieurs éléments utilisent focus:outline-none sans remplacement visible sur tous les états. Bien que focus:ring-2 soit présent sur la majorité des éléments, certains liens du footer et boutons du panneau accessibilité pourraient perdre leur indicateur de focus dans certains contextes de rendu.

Éléments concernés : liens des réseaux sociaux dans le footer, boutons du panneau d'accessibilité.

Correction : Remplacer tous les focus:outline-none par focus-visible:ring-2 focus-visible:ring-primary ou ajouter un style :focus-visible explicite dans le CSS.
13.8 — Animations sans respect de prefers-reduced-motion
Priorité : Mineur (WCAG 2.1 AA)

La classe .reveal (animations au scroll) est définie en CSS mais le JavaScript qui la déclenche ne vérifie pas la préférence utilisateur prefers-reduced-motion avant d'ajouter la classe active.

Élément concerné : js/scripts.min.js (animation au scroll des sections).

Correction : Dans le JS, vérifier window.matchMedia('(prefers-reduced-motion: reduce)').matches avant d'activer les animations.
8.7 — Badge d'accessibilité trompeur
Priorité : Mineur (bonne pratique)

Le footer affiche un badge "100%" à côté du lien Accessibilité. Cette auto-évaluation est trompeuse car elle n'est pas fondée sur un audit officiel et pourrait induire les utilisateurs en erreur.

Élément concerné : footer.html:118

Correction : Supprimer le badge ou le remplacer par un lien vers la déclaration d'accessibilité avec la mention "En cours d'évaluation" ou la date du dernier audit.

Duplication de requête sur ConseilVille

La vue home() effectue deux requêtes distinctes sur le même modèle :

-- Requête 1 : Agrégation SELECT COUNT(id), SUM(nb_habitants) FROM conseil_communautaire_conseilville; -- Requête 2 : Toutes les communes SELECT * FROM conseil_communautaire_conseilville;

Impact : +1 requête SQL inutile par chargement de page.

Optimisation : Calculer les stats en Python à partir de la liste déjà récupérée :
communes = list(ConseilVille.objects.all()) nb_communes = len(communes) nb_habitants = sum(c.nb_habitants for c in communes)
Absence d'index sur les champs de filtre

Les champs fréquemment utilisés dans les filtres et tris n'ont pas d'index :

  • ConseilVille.city_name — utilisé dans order_by du context processor
  • ConseilVille.slug — utilisé dans les URLs des communes
  • Journal.number — utilisé dans order_by("-number")

Impact : Tri full-table scan sur SQLite, dégradation avec la croissance des données.

Correction : Ajouter db_index=True dans les modèles :
class ConseilVille(models.Model): city_name = models.CharField(max_length=30, unique=True, db_index=True) slug = models.SlugField(max_length=50, unique=True, db_index=True) class Journal(models.Model): number = models.IntegerField(default=0, db_index=True)
SQLite en production

Le projet utilise SQLite (django.db.backends.sqlite3) comme base de données. SQLite n'est pas recommandé pour la production car :

  • Verrouillage au niveau fichier (pas de concurrence d'écriture)
  • Pas de cache de requêtes avancé
  • Limitations en lecture/écriture simultanée
  • Pas de réplication possible
Recommandation : Migrer vers PostgreSQL ou MySQL pour la production. Django facilite cette migration avec dumpdata / loaddata.

Recommandations d'optimisation

  1. Cache de page : Décorer la vue home avec @cache_page(300) pour éviter toute requête SQL pendant 5 minutes.
  2. Context processor : Utiliser ConseilVille.objects.only('city_name', 'slug').order_by('city_name') pour réduire le volume de données.
  3. Indexation : Ajouter db_index=True sur les champs de filtre et de tri.
  4. Lazy loading : Déjà partiellement implémenté (images avec loading="lazy"). À généraliser à toutes les images hors viewport.
  5. Compression : WhiteNoise est déjà configuré avec compression. Vérifier que Brotli est activé côté serveur web (nginx/Apache).
  6. CDN : Les polices Google Fonts sont préchargées — envisager de les self-hoster pour éviter la dépendance externe et améliorer le cache.
